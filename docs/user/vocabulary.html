Vocabulary

The access to the application requires a preliminary CAS authentication.

Once a user is authenticated, he gains access to the application and the actions he can perform in the application depend on its profile: simple user, department manager, administrator (see below).

Administrators

The first administrator of the application is set in the configuration (details here) and created when at database initialization.

An administrator can add as many administrators as he wants, and remove them (except himself since there must always be at least one administrator).

Admisnistrators are privileged users who can do everything. In particular, they can create/edit/delete storages and departments (see below).

Departments

The first department is created at database initialization.

A department is an entity for which some users have privileges; these users are called the managers of the department.

The privileges are:

  • edit the properties of the department (1)
  • create/edit/delete the other managers of the department (2)
  • create/edit/delete the rules of the storages managed by the department (3)

Storages and rules

The first storage of the application is created at database initialization.

A storage represents a virtual space, on which rules are set to determine the disk space that will be given to each user.

Rules are made up of:

  • a LDAP filter, that should correspond to users in the LDAP directory
  • a quota, the disk space given to the users who match the LDAP filter

The physical implementation of the storage is set on the web interface and determines how:

  • the rules of the storage will be applied
  • the user's quotas will be retrieved to be presented on the welcome page.

Each storage has a list of 'granted' departments. The users allowed to create/edit/delete the rules of a storage are:

  • the administrators of the application
  • the managers of the departments granted for the storage (see privilege 3 above).
